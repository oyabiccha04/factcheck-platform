# 開発判断ガイドライン

PROJECT_CHARTER.md を前提に、日々の開発判断で参照するための整理ドキュメント。

---

## 1. このプロジェクトの目的

### 北極星（判断に迷ったときの最上位基準）
**「予測を、ファクトで評価し、的中率で可視化する。」**

- 予測（事前の言明）を、結果（一次情報）で照合し、**検証可能な形で残す**
- **誰かを褒める／貶す／断罪する目的ではない**
- 予測と結果を混ぜず、**照合の履歴そのものを残す**

### 具体的な目的
| 観点 | 意味（開発時にどう解釈するか） |
|------|------------------------------|
| **何をやるか** | 格闘技（RIZIN/UFC等）の試合・興行に関する「予測」と「その検証ログ」を蓄積し、後から検証できる形で公開する |
| **何が重要か** | 「当たった／外れた」だけではなく、**いつ・どんな根拠で・どの形式で予測したか**、**結果後にどう検証し学びを更新したか**をデータとして残す |
| **何を目指さないか** | 断罪・誹謗・煽り・思想的裁定を主目的としない。ランキングを行う場合も、**事実指標としての的中率の可視化**に限定し、途中集計で数字を動かさない（興行クローズ時のみ）。 |

### 開発判断への落とし込み
- 機能追加時：「この変更は、予測と結果の照合履歴を検証可能な形で残すことに寄与するか？」で Yes なら筋が通っている
- UI/文言：ランキングを出す場合でも、煽り・断罪にならない中立な文言にする（例：「的中率（興行クローズ時点）」）。途中集計で数字を動かす導線は作らない
- データ設計：「主予測＝勝者方向」「副次＝KO/判定/SUB」の枠組みを崩さない

### 集計・ランキングの定義（追加）
- 的中率は **主予測（勝者方向）** の照合結果のみで計算する（副次＝KO/判定/SUB は参考情報として扱う）
- 集計対象は `prediction.exists = true` かつ結果（勝者）が確定しているものに限る
- 集計・表示は **興行クローズ時のみ**（途中集計はしない）
- **混在興行の種目スコープ（2026-03 追記）**：RIZIN 等で1ファイルに MMA / KICKBOXING が混在しうる。`martial-arts/predictors.html`（MMA）および `kick-predictors.html`（キック）の **予測者ランキング・検索用の名前抽出**では、`cards[].bouts[].discipline` がページ種目と一致する bout のみを対象にする。`statements[]` は `prediction.target` を red/blue の対戦表記と正規化マッチし、対応 bout の `discipline` で判定する。データ追加時は **bout に正しい `discipline` を必ず付与**し、`target` はカード表記と揃える（集計の前提）。

### 予測メタ情報（キー名の固定・追加）
- 予測者は `predictor_name`（string）を正とする。未確定・未定義の場合は **"不明"** と明示する（推測で補完しない）
- **同一人物は同じ表記に統一する**：同一の予測者（人・媒体）は、興行をまたいでも **必ず同じ `predictor_name` 表記** を用いる。表記ゆれ（例：略称と正式名の混在）は避け、方針（媒体名で揃える／本名で揃えるなど）を決めて運用する。集計・ランキングの一貫性のため必須とする。
- 予測ソースは `source_url` / `source_ref` / `source_note` のいずれかで一次情報に辿れる形にする。不明は **"不明"** と明示する
- **信頼性の優先（一次情報の扱い）**：
  - `source_url` は **一次情報** を優先して記載する。一次情報＝公式発表・公式動画・公式サイト・新聞社の原記事など、根拠が検証可能な一次ソースとする。
  - 予測抽出では **日本国内の予測・一次情報を優先** する。日本語記事/動画/SNS、日本国内メディア、日本で活動する発信者の一次情報を先に当たる。
  - 同条件で比較できる場合は海外ソースより国内ソースを優先し、国内だけで件数要件を満たせない場合のみ海外ソースで補完する。
  - 個人ブログ・まとめサイト・SNSまとめなど **二次情報のみ** の場合は、`source_note` に「※二次情報」と一次情報の所在（例：公式YouTube、URL未同定）を明記し、一次情報のURLが判明し次第 `source_url` を差し替える。
  - データ追加時は、可能な限り一次情報を特定してから登録する。
  - YouTube 動画の引用文やタイムスタンプ抽出が必要な場合、補助的に Gemini 等の外部AIを利用してもよい。その場合は `source_note` に「Gemini補助で抽出（確認日：YYYY-MM-DD）」等のメモを残す。
  - 発言者や勝者方向が一次情報だけで確定できない場合は、推測で埋めず **"不明"** と明示し、`source_note` に「要確認」と未確定理由・根拠を残す。

### Gemini出力の最低検査（運用メモ・追加）
- Gemini等の外部AI出力は、そのまま取り込まず **一次URLの妥当性を必ず検査**する（架空出典の混入を防ぐ）。
- **次に該当するレコードは破棄/再抽出**（推測で修正しない）：
  - URLが `sample` / `example` / ダミーっぽいIDを含む（実在性不明）
  - URLが Markdown 形式（`[text](url)`）になっている（生URLに直しても元がダミーの場合は破棄）
  - `target` が当該イベントの公式カードと一致しない（別興行/別カード混入の可能性。一次情報で再確認する）

### 観測する予測者（対象範囲の定義・追加）
- **有名YouTuberを観測対象に含める**
  - 「有名」の定義：**登録者数 10 万以上**
  - 判定が曖昧な場合は、`source_note` に根拠メモ（チャンネル名、確認日、スクショURLなど）を残す
- `predictor_name` は **個人名で統一**する
  - 1本の動画に複数人が出演して予測する場合があるため
  - その場合は **人物ごとに予測レコードを分け**、`source_url` を共有しつつタイムスタンプ等で区別する

## 2. 設計思想

### 思想の核
- **予測を評価するためではなく、人を格付けするためでもなく、正しさを裁定するためでもない**
- **予測と結果を分離し、照合可能な形で記録し続けること——その行為そのものに価値がある**
- 感情や権威ではなく **事実を見る態度（FACTFULNESS的）** と整合する。照合結果を **的中率** として可視化し、予測者別ランキングを表示することがあるが、断罪・誹謗・煽りを目的としない

### コンテンツの位置づけ
- 格闘技は **「予測→検証→事実の蓄積」という枠組みを、他分野へ拡張可能な形で実証するための最初の題材**
- 政治・経済・社会事象など他分野へ拡張する場合も、北極星と本憲法の原則は厳守する
- **感情的評価・思想的断罪・支持・不支持の表明を主目的とするコンテンツ化は行わない**

### 収益化の位置づけ
- 広告・寄付による収益化は目指すが、**収益化は以下を正当化する理由にはならない**
  - 予測や結果の改変
  - 特定の結論への誘導
  - スポンサー・支援者への迎合
- **予測と結果の分離・一次情報に基づく事実の保持・履歴の改変を行わないこと** を最優先する
- 収益は「サイト運営・データ整備・長期的な履歴保存」の手段であり、**プロダクトの思想そのものではない**

### 開発判断への落とし込み
- 新機能：「人を評価するUI」より「照合履歴が追いやすいUI」を選ぶ
- 収益機能：「結論を誘導する表示」や「スポンサーに合わせた表示のゆがみ」は禁止とみなす
- 他分野拡張時：格闘技で確立した「1試合1JSON」「全文更新」などの型を崩さない

## 3. 優先すべきこと

### データ・運用（最優先で守る）
| 項目 | 具体的なルール | 開発時の意味 |
|------|----------------|--------------|
| **正本** | セーブデータ完全統合版が唯一の正本 | データは「全文更新」のみ。差分更新・簡略化は **禁止** |
| **データ単位** | 1試合＝1 JSON（standard-log-v1.2-json） | 興行単位でまとめる場合は「1興行＝1ファイル」に統一。**試合単位と興行単位の混在は禁止** |
| **予測の扱い** | 主予測＝勝者方向、副次＝KO/判定/SUB | この枠組みを崩さない範囲で拡張する |
| **集計** | 予測者別集計は `predictor_name` が明記された予測を対象、興行クローズ時のみ | 途中集計で数字を動かさない |
| **予測ソース（出典）** | 予測には必ず一次情報への参照を持たせる（URL/タイムスタンプ/参照ID等）。不明なら「不明」と明示 | 信頼性担保のため、読者が根拠に遡れる形を維持する |
| **スコープ** | 会場 → スコープの順で判定、一度決めたら逆転不可 | 実装でも「後からスコープをひっくり返す」ロジックは持たない |
| **ドームクラス** | 東京ドーム開催のみ（独自定義） | 他会場を「ドームクラス」に含めない |
| **MVP** | indexes 作成禁止 | MVP 完了まではインデックス機能は作らない |

### ページ・技術
- **表示の基準ページは logs.html**（logs_v2.html があっても運用基準は logs.html）
- トップ `index.html` は一般向けに寄せ、**`data/index.json` 直リンク・リポジトリ・イベント raw JSON** の導線は **`verify.html`** にまとめる（実装メモ・2026-03）
- トップの構成順：**入口 → About（要約）→ 最新イベント（日付降順のみ）→ 使い方**。使い方では「予測ハブは入口」「ログ一覧は各予測ページから」と明示する（2026-03-24）
- データは **「表示ページが読み込める場所」** に置く
- 推奨ディレクトリ：`data/`（興行・試合JSON）、`master/`（定義・参照データ）
- Master Definition：`/master/master_definition.json` を前提にした参照にする

#### 予測者ハブ（一覧・ランキング・検索）
| 種目・領域 | 主なHTML | 備考 |
|------------|----------|------|
| MMA（格闘技） | `martial-arts/predictors.html` | `data/index.json` + 各イベントJSON。詳細は `martial-arts/predictor.html?name=…` |
| キック・ムエタイ | `kick-predictors.html`（リポジトリ直下） | キック導線の正。詳細は上と同じ `martial-arts/predictor.html` |
| ボクシング | `predictors.html`（直下） | 詳細は直下 `predictor.html` |
- いずれも予測者一覧は **自動生成**（手書き更新しない）。`"不明"` 等は一覧ノイズ低減のため除外し、検索では件数表示など既存UIに合わせる。

#### グローバルナビ（2026-03 追記）
- 主要ページ共通：**トップ** / **MMA予測** / **ボクシング予測** / **キック予測** の4項目のみ（ドロップダウンでログ一覧を出さない）
- リンク先は上表の3 predictors ハブ + トップ。`martial-arts/` 内のページでは `../` で直下の `predictors.html` / `kick-predictors.html` に繋ぐ

### 「充実」の定義（何を優先して作るか）
このサイトで充実とは、次の 4 つを満たすこと：
1. **初見の人が意味を理解できる**（何のサイトか／何が見られるか）
2. **同じ基準で比較できる**（興行・試合をまたいで形式が統一されている）
3. **検証できる**（予測→結果→振り返りがつながる）
4. **更新が続く**（運用が苦しくならない）

### 開発判断への落とし込み
- 仕様が曖昧なとき：上記 4 項目のどれに効くかを明らかにしてから実装する
- データ形式を変えるとき：他興行・他試合と「同じ基準で比較できる」かを必ず確認する
- 運用負荷が増える変更：「更新が続く」を損なわないか（自動化・テンプレ化・手順の明確化）を検討する

## 4. やってはいけないこと

### データ・運用（禁止事項）
| 禁止内容 | 理由・開発時の注意 |
|----------|-------------------|
| **差分更新・簡略化** | 正本は全文更新のみ。部分だけ書き換える運用は禁止 |
| **試合単位と興行単位の混在** | 1リポジトリ内で「このファイルは試合単位」「あのファイルは興行単位」が混ざらないようにする |
| **途中集計で数字を動かす** | 集計は興行クローズ時のみ。実装でも「途中集計モード」を正規の仕様にしない |
| **ソース不明を推測で補完する** | 一次情報に基づく事実の保持が最優先。ソースがない予測は「不明」と明示し、推測で埋めない |
| **スコープ判定の逆転** | 一度「会場→スコープ」で決めたら後からひっくり返さない（データ・UIとも） |
| **MVP での indexes 作成** | MVP 段階では indexes 機能を追加しない |

### コンテンツ・表現（禁止事項）
| 禁止内容 | 理由・開発時の注意 |
|----------|-------------------|
| **断罪・誹謗・煽り・支持/不支持の表明を主目的とするコンテンツ** | 北極星に反する。ランキングを表示する場合でも、事実指標（的中率）の提示に限定し、人格批判や断定的な評価表現にしない |
| **予測や結果の改変** | 一次情報と事実の保持が最優先。編集機能があっても「改変の履歴が残らない」形にしない |
| **特定の結論への誘導** | 表示順・強調・非表示で「ある結論を押し出す」ような設計にしない |
| **スポンサー・支援者への迎合で表示やデータを曲げる** | 収益化は思想を曲げる理由にならない |

### AI 依頼・進め方（推奨の反対＝避けること）
| 避けること | 代わりにすべきこと |
|------------|-------------------|
| **大きな依頼を一気に出す** | 小さな単位に切って依頼する |
| **PROJECT_CHARTER を参照せずに設計する** | 前提は PROJECT_CHARTER.md（本ガイド含む）を毎回参照する |
| **データ形式を Standard Log v1.2 系以外で提案する** | データ形式は Standard Log v1.2 系に合わせる |
| **logs.html 以外を無断で本番基準にする** | 表示の基準ページは logs.html とし、変更時は文書と合意する |

### 開発判断への落とし込み
- 新規画面：「誰が当たったか」を前面に出すより「どの予測がどう照合されたか」を前面に出す
- 編集機能：「結果を後から書き換え可能だが履歴が残らない」は禁止レベルで避ける
- 収益系：「この表示はスポンサー都合で変えていい」という仕様にしない

## 5. クイックチェック（開発前・PR前）

- [ ] 北極星「予測を、ファクトで評価し、的中率で可視化する」に照らして、断罪・誹謗・煽りの機能・文言になっていないか（ランキングは事実指標としての提示に限定できているか）
- [ ] データは全文更新の方針を守り、差分更新・簡略化をしていないか
- [ ] データ単位は「1試合1JSON」または「1興行1ファイル」で統一し、混在していないか
- [ ] 表示の基準は logs.html とし、参照パスは「表示ページが読み込める場所」に合わせているか
- [ ] 新規HTMLを追加する場合、**グローバルナビ**を既存主要ページと同じ4リンク（トップ / MMA予測 / ボクシング予測 / キック予測）に揃えているか
- [ ] 混在興行JSONで bout に **`discipline`** を付け、`statements` の **`target`** をカード表記と揃えているか（種目別ランキングの前提）
- [ ] データ再利用・商用条件・免責を変えた場合、`policy.html` の文言を更新し、about / verify からの導線が壊れていないか
- [ ] 収益・スポンサーを理由に、予測・結果の改変や結論の誘導をしていないか
- [ ] MVP の範囲で indexes を新規作成していないか（該当する場合のみ）
